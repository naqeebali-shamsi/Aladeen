## 2026-04-20 ‚Äî runaway-retry-loop

- **Runs:** `690cbbfe-a2e5-4a07-aedf-7d415d417e4d` (`.aladeen/runs/690cbbfe-a2e5-4a07-aedf-7d415d417e4d.json`),
  `b9428fa6-17d4-42ea-91db-85c62f593e74` (`.aladeen/runs/b9428fa6-17d4-42ea-91db-85c62f593e74.json`)
- **Blueprint:** `implement-feature-deterministic`
- **Node:** `lint` (deterministic, attempts: 1930 / 2267 respectively)
- **Evidence:** Same `spawn node ENOENT` failure repeated thousands of times; `fix-lint` then re-wrote the same file and looped back. `totalRetries` stayed at 0 because the runner only bumped it inside the agentic-retry path. `maxTotalRetries: 3` and `maxDurationMs: 120000` were configured but never tripped ‚Äî the budget never saw the loop.
- **Suggested next action:** Increment `totalRetries` on every node re-entry so deterministic loops count toward the global budget.
- **Status:** resolved (commit `582e830`) ‚Äî regression test in `src/engine/runner.test.ts`

## 2026-04-20 ‚Äî worktree-dependency-miss

- **Runs:** `690cbbfe-a2e5-4a07-aedf-7d415d417e4d`, `b9428fa6-17d4-42ea-91db-85c62f593e74`
- **Blueprint:** `implement-feature-deterministic`
- **Node:** `lint` (deterministic, attempts: 1930+)
- **Evidence:** `Cannot find module 'N:\Aladeen\.aladeen\worktrees\e2e-test-001\node_modules\typescript\lib\tsc.js'`. Underlying cause of the runaway loop: `git worktree add` does not bring `node_modules`, so the tsc invocation can never resolve.
- **Suggested next action:** WorktreeManager should opportunistically junction `repoRoot/node_modules` into each new worktree.
- **Status:** resolved (commit `99b76d4`) ‚Äî `WorktreeManager.linkNodeModulesIfPresent()`. Validated end-to-end by rerunning the same blueprint: 1.8s success vs prior infinite hang.

## 2026-04-20 ‚Äî orphaned-running-state

- **Runs:** `690cbbfe-a2e5-4a07-aedf-7d415d417e4d`, `b9428fa6-17d4-42ea-91db-85c62f593e74`
- **Blueprint:** `implement-feature-deterministic`
- **Node:** N/A (run-level)
- **Evidence:** Both runs still have `status: "running"` in their persisted state files weeks after the user killed them. `currentNodeId: "lint"` is also stuck. The runner has no liveness/heartbeat that lets a fresh process detect "this run is orphaned."
- **Suggested next action:** On runner startup or on `list-runs` invocation, sweep `.aladeen/runs/*.json` for `status: "running"` with `startedAt` older than `maxDurationMs * 2` (default ~20min) and rewrite to `status: "abandoned"` so they don't pollute future analysis.
- **Status:** open
